## 0. Экосистема разработки серверов SCP: SL

Чтобы корректно оценить FYDNE, фиксируем целевую экосистему — именно её сдвиги ломают проект.

**Базис движка.** Выделенный сервер SCP: SL — headless-сборка на Unity (Mono, рантайм, совместимый с `.NET Framework 4.8`). Вся игровая логика — в сборках `Assembly-CSharp.dll` и `Assembly-CSharp-firstpass.dll`. Сетевой слой — **Mirror** (форк UNET). Голос — отдельный стек (`VoiceTransceiver`, Opus). Ключевая проблема модинга: Northwood **не предоставляет стабильного публичного API на уровне `Assembly-CSharp`** — внутренние типы, поля и сигнатуры методов переименовываются и реструктурируются почти каждый крупный апдейт.

**Слои фреймворков (исторически):**

1. **NW Plugin API (NWAPI / PluginAPI)** — официальный, но «тонкий» и нестабильный API от Northwood (~2021–2024).
2. **EXILED** — доминирующий community-фреймворк: событийная модель на делегатах (`Exiled.Events.Handlers`), обёртки `Exiled.API.Features.Player/Map/Round`, загрузчик поверх Harmony.
3. **LabAPI** — новый официальный фреймворк Northwood (конец 2024), унифицирует и вытесняет NWAPI; EXILED мигрировал на него как на фундамент. Именно `LabApi.dll` присутствует в зависимостях `Loli-Merged`.
4. **Qurre** — сторонний (русскоязычный) фреймворк-альтернатива EXILED, на котором построен FYDNE. Событийная модель **атрибутная**: `[EventMethod(PlayerEvents.Spawn, priority)]` вместо делегатов EXILED.

**Инструмент рантайм-патчинга** во всех фреймворках — **Lib.Harmony** (`0Harmony.dll`): префиксы/постфиксы/транспайлеры поверх методов `Assembly-CSharp`. Это и есть главная точка хрупкости: транспайлер привязан к конкретной структуре IL-кода целевого метода, и при его изменении в апдейте патч либо падает на старте, либо ломает геймплей.

## 1. Архитектура и структура проекта

FYDNE — не «плагин», а **распределённая платформа** из трёх контуров: игровой плагин (C#), бэкенд-микросервисы (Node.js) и связующий сокет-протокол.

### 1.1. Игровой контур — `Loli-Merged`

Главный геймплейный плагин (внутреннее имя `Loli`, версия в коде `6.6.6`).

- **Целевой фреймворк сборки:** `net48`, `LangVersion 12.0`, `AllowUnsafeBlocks`. csproj адаптирован под кросс-сборку на Linux через Mono (`FrameworkPathOverride=/usr/lib/mono/4.5/`, mono-api пути в `AssemblySearchPaths`) — прод собирался/работал на Linux.
- **Зависимости (из `.csproj`):** `Qurre`, `QurreSocket`, `LabApi`, `Assembly-CSharp(_public)`, `Assembly-CSharp-firstpass`, `Mirror(_public)`, `Mirror.Components`, `0Harmony`, `CommandSystem.Core`, `NorthwoodLib`, `Pooling`, `Newtonsoft.Json`, `SchematicUnity`, `SCPLogs`, `Unity.TextMeshPro`, `MEC` (Timing). HintPath ведут на внешние каталоги разработчика (`..\..\Qurre-sl\...`, `..\..\..\shared-termius\...`) — **в репозитории нет самих DLL зависимостей**, проект из коробки не собирается.
- **Точка входа** — `Core.cs`: статический класс с атрибутами `[PluginInit]`/`[PluginEnable]`/`[PluginDisable]`. В `Enable()`:
  - открывается `QurreSocket.Client` на `:2467` к хардкоженному IP бэкенда;
  - регистрируются обработчики сокет-событий (`token.required`, `connect`);
  - запускаются корутины MEC (`UpdateVerkey` и др.);
  - регистрируются RA-команды;
  - `new Harmony("fydne.loli").PatchAll()` — массовое применение Harmony-патчей.
  - `Disable()` сводится к `Server.Restart()` — корректной выгрузки плагина нет, рассчитано на полный рестарт процесса.
- **Два build-flavor через `#if`:** `MRP` (Medium RolePlay) и `NR` (NoRules). Серверы (ServerID 1–4: Medium RP, NoRules, Friendly NoRules, AdmZone) разводятся по `Server.Port` (7666/7779/6666/7888) и директивам компиляции — **132 блока `#if MRP` и 96 `#if NR`**. Это монолит с компиляцией под конкретный сервер, а не конфигурируемый рантайм.

**Модульная декомпозиция** (`Loli/`, ~196 `.cs`, ~27k LOC):

| Каталог | Назначение |
|---|---|
| `Addons/` | Чат, хинты, авто-модерация (`Executor.cs`, 408 строк), RolePlay-механики (рероворк SCP-173/096, реалистичные наручники), спавн, аирдроп, команды (`Commands.cs`, 1022 строки) |
| `Concepts/` | Кастомные режимы/механики: CO2, Ragnarok (Priest), Hackers (MainServers/Control), NuclearAttack, Scp008/ControlRoom |
| `Builds/Models/` | Постройки на схематиках (`SchematicUnity`): лифты, комнаты, серверная, AdminRoom |
| `DataBase/` | Слой персистентности: уровни (`Levels.cs`, 614 строк), донат, кастомизация, загрузчик данных через сокет |
| `Scps/`, `Spawns/` | Доп. логика SCP (079Better, 294) и спавн МТФ (`MobileTaskForces.cs`, 564 строки) |
| `HintsCore/`, `Modules/`, `Patches/` | Система хинтов, голоса/Cassie, Harmony-патчи (фиксы и блокировки гейм-механик) |
| `Webhooks/`, `Logs/` | Discord-вебхуки, логирование |

**Событийная модель** — атрибутная (Qurre). Частота подписок: `RoundEvents.Waiting` ×66, `PlayerEvents.Spawn` ×33, `RoundEvents.Start` ×20, `PlayerEvents.ChangeRole` ×16, `PlayerEvents.Dead/Join` и т.д. Приоритеты — вторым аргументом (`int.MinValue/MaxValue/666/-2`).

### 1.2. Бэкенд-контур — `fydne-prod-web-2020-2023`

Node.js, менеджер процессов **pm2** в режиме `cluster`. **14 сервисов** (из `pm2.config.js`):

| `web connect` | `web-connect/connect.js` | Привязка аккаунтов (Steam/Discord) |
| `web socket` | `web-socket/init.js` | Realtime для сайта |
| `web clans socket` | `web-clans-socket/init.js` | Клановые войны |
| `cdn` | `cdn/cdn.js` | Раздача ассетов (`cdn.loliscp.ru`), imagemin |
| `manager` | `bots/manager` | Discord/TG-боты |

Состояние стека по `package.json`: `@sentry/node@^9.9` (релиз 2025), `mongoose@7/8`, `express@4.18`, `socket.io@4` — **бэкенд относительно современный, поддерживался вплоть до окт-2025**.

### 1.3. Связующий слой — кастомный протокол `qurre-socket`

Собственная Socket.IO-подобная обёртка (`Server`/`Client`, npm-пакет `qurre-socket`, C#-аналог `QurreSocket.dll`). Контракт — событийный, по строковым каналам.

Каталог сообщений из плагина (`Socket.Emit/On`):

- **Аутентификация:** `SCPServerInit` (отдаёт `ApiToken`), `token.required`.
- **Присутствие/IP:** `server.addip`, `server.clearips`, `server.join`, `server.leave`, `server.tps`.
- **Запросы к БД (Emit→On пары):** `database.get.stats`, `database.get.data`, `database.get.donate.{roles,ra,customize}`, `database.get.nitro`, `database.get.patrol`, `database.get.adm.steams`, `database.add.time`, `database.remove.admin`, `database.admin.kick`, `server.database.clans` → `socket.database.clans`, `database.internal.unsafe.set_level`.

Загрузка данных (`DataBase/Modules/Loader.cs`) — асинхронный RPC: плагин делает `Emit(запрос)`, бэкенд отвечает `On(ответ)`, callback парсит `object[]` и **сырыми `new Thread(...)`** наполняет внутренние структуры. Сериализация нетипизированная (`((IEnumerable)data[0]).Cast<object>().Select(x => x.ToString())`).

**Связь контуров:** игровой сервер (C#) — клиент `:2467` → `scp-socket` (Node) — единственная точка записи в MongoDB → остальные web-сервисы читают ту же Mongo. Сайт и игра не общаются напрямую, всё проходит через Mongo + сокет-хаб.

## 3. Технический аудит ошибок

1. **Hard-coupling с непубличным игровым API.** ~40 `[HarmonyPatch]` на внутренние типы + 12 транспайлеров. Патчатся: `FirearmDamageHandler.ProcessDamage`, `VoiceTransceiver.ServerReceiveMessage`, `Scp079Recontainer.OnServerRoleChanged/PlayAnnouncement`, `Scp173BlinkTimer.AbilityReady`, `Scp3114Strangle.ServerProcessCmd/ServerUpdateTarget`, `Scp3114Spawner.OnPlayersSpawned`, `Scp956Pinata.Update`, `Scp2536GiftController.TryGrantWeapon`, `DeadmanSwitch.OnUpdate`, `NineTailedFoxAnnouncer.Update`, `CandyPink.ServerApplyEffects`, `Adrenaline.OnEffectsActivated` и др. Набор типов (Scp3114/Scp956/Scp2536) указывает на игру эпохи **v13–v14 (конец 2024 – 2025)**. Любое переименование/смена сигнатуры → `PatchAll()` падает целиком и роняет загрузку всего плагина.

2. **Транспайлеры без валидации входного IL.** Пример (`Patches/AntiCringeUpdates.cs`):
   ```csharp
   [HarmonyPatch(typeof(Scp3114Strangle), nameof(Scp3114Strangle.ServerProcessCmd))]
   static IEnumerable<CodeInstruction> Call(IEnumerable<CodeInstruction> instructions)
   {
       yield return new CodeInstruction(OpCodes.Ret);   // полностью вырезает тело метода
   }
   ```
   Транспайлер игнорирует входной IL и подменяет метод на `return`. Сломается при изменении сигнатуры/возвращаемого типа (несбалансированный стек → `InvalidProgramException`), а семантически отключает штатную логику игры — после апдейтов даёт трудноуловимые баги (рассинхрон, эксплойты).

3. **Зависимость от Qurre — single-point-of-failure.** Qurre — сторонний NuGet (в архиве `Qurre.1.14.2`), community-проект одного мейнтейнера. Нет сборки под текущую игру/LabAPI — весь FYDNE мёртв. Параллельное использование `LabApi` + `Qurre` создаёт риск двойной обёртки над одними игровыми объектами (рассинхрон состояния Player/Map).

4. **Монолит с компиляцией под сервер (`#if MRP/NR`).** 228 директив условной компиляции делают невозможной единую сборку и усложняют поддержку.

5. **Нетипизированный сокет-RPC.** Контракт на `object[]` и `.ToString()`, без схемы/версионирования. Расхождение порядка/типов полей → молчаливые `catch {}` (исключения проглатываются), данные просто не грузятся, без диагностики.

6. **Потоковая модель «сырыми `Thread`».** Загрузка данных стартует `new Thread(...)` из сокет-callback и пишет в общие коллекции — гонки с однопоточным игровым потоком Unity. Латентные баги под нагрузкой.

7. **Отсутствие воспроизводимой сборки.** Зависимости — по абсолютным/относительным путям к чужим каталогам, без NuGet-restore и без vendored-DLL. Нет CI, нет lock на версии игровых сборок.

8. **Безопасность.** В `Core.cs` зашиты прод-секреты: `ApiToken`, `SteamToken` (Steam Web API key), IP бэкенда, домены. Репозиторий публичный → **все ключи скомпрометированы**. Повсеместные `catch {}` скрывают ошибки.

## 5. План восстановления (вектор разработки)

### 5.1. Что чинится минимальными правками

- **Бэкенд-контур (Node).** Поднять MongoDB, восстановить `config.js` каждого сервиса (Mongo URI, `ApiToken`, IP), убрать/дописать 3 отсутствующих сервиса в `pm2.config.js`, сгенерировать **новые** `ApiToken`/`SteamToken`. Рабочая основа без переписывания.
- **Чистая логика плагина** (экономика, уровни, кастомизация, вебхуки, сокет-клиент) — переносится почти как есть.
- **Транспайлеры-заглушки → префиксы.** Замена `yield return Ret` на `[HarmonyPrefix] static bool P() => false;` снимает класс крашей из-за несбалансированного IL.
- **Конфигурация из кода в файлы.** Вынести секреты/порты/ServerID из `Core.cs` и `#if` в JSON (механизм `JsonConfig` уже есть).

### 5.2. Что критически устарело и требует переработки с нуля

- **Слой совместимости с игрой (`Patches/` и прямые `[HarmonyPatch]`).** Привязан к v13–v14. Ревизия «метод за методом» под текущую `Assembly-CSharp` и **миграция на стабильные события LabAPI/EXILED** там, где есть эквивалент (урон, роли, голос, SCP-абилки).
- **Зависимость Qurre.** Стратегическое решение: либо собрать актуальный Qurre под текущую игру, либо **мигрировать FYDNE на LabAPI/EXILED**. Учитывая, что `LabApi.dll` уже в зависимостях, миграция на LabAPI — наиболее жизнеспособный путь.
- **Монолит `#if MRP/NR`.** Рефактор в единую сборку + рантайм-конфиг профилей серверов.
- **Сокет-протокол.** Типизация и версионирование сообщений, замена `catch {}` на логирование, отказ от сырых `Thread` в пользу диспетчеризации в главный поток Unity (`MEC.Timing`).

### 5.3. Рекомендуемый порядок воскрешения

1. **Инвентаризация зависимостей.** Зафиксировать точную версию игры (по типам — v13/v14), собрать/найти `Assembly-CSharp`, `Qurre`, `LabApi`. Завести как vendored-DLL, переписать HintPath на относительные. Цель — воспроизводимая сборка.
2. **Поднять бэкенд** (Mongo + pm2) с новыми секретами — рабочий персистент-контур для тестов.
3. **Собрать плагин под зафиксированную версию**, прогнать `PatchAll()`, составить список падающих патчей. Превратить «не работает» в конкретный перечень дефектов.
4. **Чинить патчи по приоритету:** критичные (урон, роли, голос, recontain SCP-079) — на события LabAPI; заглушки — на префиксы.
5. **Стратегическое решение по фреймворку** (Qurre vs LabAPI) — до массовой починки, чтобы не чинить дважды.
6. **Усиление надёжности:** убрать хардкод-секреты, типизировать сокет-протокол, заменить `catch {}` на логи, навести потокобезопасность.

## 6. Вывод: восстанавливать или с нуля

Ответ **разный для каждого контура** — это ключевой момент.

### Бэкенд (Node-микросервисы) → однозначно ВОССТАНАВЛИВАТЬ
Архитектура зрелая, стек свежий, код целый. Писать заново 14 сервисов, Mongo-схемы, сокет-протокол и сайт — месяцы впустую. Восстановление = Mongo + конфиги + перевыпуск секретов + убрать 3 отсутствующих сервиса. **Реалистично: 1–2 недели.**

### Игровой плагин → ГИБРИД: каркас заново, механики — донорство
Полное восстановление упирается в стену: **критичных зависимостей для сборки в репо нет**, а сам код жёстко прибит к v13–v14 через 40 патчей и 12 IL-транспайлеров. Восстановление «как было» = бесконечная гонка за патчами Northwood на чужом single-maintainer-фреймворке.

- **С нуля:** инфраструктурный каркас (точка входа, lifecycle, событийка, сокет-клиент) — на **LabAPI** (официальный, поддерживаемый), а не на связке Qurre + прямые патчи.
- **Восстанавливать (донорство):** ~70% контента (уровни, донат, кастомизация, кланы, кастомные режимы, вебхуки) — портировать из существующего кода. Самая трудоёмкая часть уже написана.

### Итоговая таблица решения

| Контур | Решение | Оценка трудозатрат |
|---|---|---|
| Бэкенд Node | **Восстановить как есть** | 1–2 недели |
| Сокет-протокол | **Восстановить** (+ типизация/версионирование) | 2–4 дня |
| Плагин: каркас/игровой слой | **Переписать с нуля на LabAPI** | основная работа, недели–месяцы |
| Плагин: контент-логика | **Портировать из репо** (донорство) | экономит ~70% объёма |
| Архив плагинов | **Держать как библиотеку-референс** | — |

**Короткий ответ:** легче и правильнее **не «восстанавливать целиком» и не «писать всё с нуля», а гибрид.** Бэкенд и весь контент-код — восстанавливать (всё уже написано и работоспособно). Игровой слой плагина — пересобрать каркас заново на LabAPI, портируя готовые механики. Дословная реанимация `Loli-Merged` на Qurre снова умрёт после ближайшего апдейта игры и физически не собирается из-за отсутствующих зависимостей.

## 7. Требования к серверной инфраструктуре

Две принципиально разные нагрузки: **игровые инстансы (упираются в CPU)** и **бэкенд (упирается в RAM)**. Их разносят.

**Что показывает конфигурация:**
- Игровые серверы — **4 инстанса** (Medium RP, NoRules, Friendly NoRules, AdmZone), слотов по умолчанию `max_players = 100`.
- Бэкенд — 14 pm2-сервисов, суммарный потолок памяти (`max_memory_restart`) **≈ 6.4 GB**, плюс **MongoDB** отдельным демоном (самый прожорливый по RAM при росте базы).

**Ключевой факт:** выделенный сервер SCP: SL **однопоточный** — вся игровая логика в одном Unity-потоке. Решает **частота одного ядра**, не их количество. FYDNE с ~40 Harmony-патчами, корутинами и схематиками нагружает поток сильнее ванили. NoRules на 100+ игроков — самый тяжёлый кейс.

### Требования по сценариям

**1. Dev / тест (1 сервер + бэкенд на одной машине):**

| Ресурс | Минимум |
|---|---|
| CPU | 4 ядра, ≥3.5 ГГц (важна частота) |
| RAM | 8 GB |
| Диск | 40 GB SSD |
| ОС | Linux (проект собирался под Mono/Linux) |

**2. Одиночный прод (1 игровой инстанс + полный бэкенд):**

| Ресурс | Рекомендация |
|---|---|
| CPU | 6–8 ядер с высокой частотой (Ryzen 7/9, EPYC, 4.0–4.5 ГГц+) |
| RAM | 16 GB |
| Диск | 80–120 GB NVMe SSD |
| Сеть | 1 Гбит/с, low-latency (голос/Mirror чувствительны к пингу) |

**3. Полный прод как было (4 игровых сервера + бэкенд + Mongo), 2 машины:**

| Узел | CPU | RAM | Диск |
|---|---|---|---|
| **Игровой узел** (4 инстанса SCP:SL) | 8–12 быстрых ядер (4.5 ГГц+), приоритет single-thread | 24–32 GB (~3–4 GB на инстанс с плагином) | 100 GB NVMe |
| **Бэкенд-узел** (14 pm2-сервисов + MongoDB) | 4–8 ядер | 16–32 GB (6.4 GB сервисы + Mongo working set + запас) | 100–200 GB NVMe |

На одной машине ориентир: **12+ быстрых ядер, 48–64 GB RAM, 200 GB NVMe, 1 Гбит/с.**

**Итог:** для воскрешения/разработки хватит **4 ядра / 8 GB / SSD**. Для прод-онлайна главное не «мощность вообще», а **высокая частота ядра** (однопоточность SCP:SL) под игровые инстансы и **много RAM под Mongo + боты** на бэкенде. Точный объём RAM под MongoDB зависит от размера накопленной базы (её в репо нет — закладывать запас).
